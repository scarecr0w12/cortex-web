# Sandbox Guide

CortexPrism provides sandboxed code execution using Docker containers (with subprocess fallback). This guide covers usage, configuration, and security.

## Overview

The sandbox system allows agents to execute code safely in isolated environments with resource constraints. Code runs in ephemeral containers that are destroyed after execution.

## How It Works

```bash
cortex sandbox run script.py                 # Auto-detect language, run in sandbox
cortex sandbox run script.py --no-sandbox    # Skip sandbox, run as subprocess
cortex sandbox run script.py --fix           # Enable auto-fix on failure
```

## Docker Sandbox

When Docker is available, containers are created with strict resource limits:

| Constraint | Value |
|------------|-------|
| Network | `--network=none` (no external access) |
| Memory | 256MB limit |
| CPU | 0.5 cores |
| Process limit | 64 PIDs |
| Security | `no-new-privileges` |
| Timeout | 30 seconds |
| Max output | 64KB |

## Supported Languages

| Language | Extension | Docker Image | Interpreter |
|----------|-----------|-------------|-------------|
| Python | `.py` | `python:3.12-alpine` | `python3` |
| JavaScript | `.js` | `node:22-alpine` | `node` |
| TypeScript | `.ts` | `denoland/deno:alpine` | `deno eval` |
| Bash | `.sh` | `alpine:3.20` | `bash` |
| Ruby | `.rb` | `ruby:3.3-alpine` | `ruby` |
| Go | `.go` | `golang:1.22-alpine` | `go run` |
| Rust | `.rs` | `rust:1.78-alpine` | `rustc -o /tmp/out && /tmp/out` |

## Subprocess Fallback

When `docker info` fails (Docker not installed or daemon not running), the sandbox falls back to subprocess mode. The code runs directly on the host machine with the same resource limits applied at the process level.

## Security Considerations

- Docker sandbox has **no network access** (`--network=none`)
- Memory and CPU are strictly limited to prevent resource exhaustion
- Containers are ephemeral â€” no data persists after execution
- The `no-new-privileges` security flag prevents privilege escalation
- Output is capped at 64KB to prevent log flooding

## Configuration

Sandbox resource limits are configurable in `~/.cortex/config.json` under the `sandbox` section (see [Performance Tuning](/docs/knowledge-base/performance-tuning#sandbox-resource-limits) for details):

| Constraint | Value |
|------------|-------|
| Memory limit | 256MB |
| CPU limit | 0.5 cores |
| PID limit | 64 |
| Execution timeout | 30 seconds |
| Max output | 64KB |

2. **Enable auto-fix** during development to iterate faster
